# Critical Issues 🔴

### 🔴 C-1 — Live `/robots.txt` is a static file that contradicts the code and inverts intent
**What:** The dynamic route `app/robots.ts:19` returns `User-agent: *` / `Disallow: /` for any non-production host (correct: block *everyone* on staging). But the **live** `/robots.txt` returns something different:

```
$ curl -i https://darkviolet-gerbil-901490.hostingersite.com/robots.txt
content-type: text/plain   content-length: 58   server: hcdn
User-agent: Googlebot
Disallow: /

User-agent: *
Allow: /
```

This is **not** what `app/robots.ts` produces — it blocks *only Googlebot* and **explicitly allows every other crawler** (Bingbot, GPTBot, ClaudeBot, PerplexityBot, etc.) on the staging domain.
**Root cause:** It is being served as a **static file by Hostinger's edge (`server: hcdn`, `text/plain`, fixed `content-length: 58`)**, not by Next.js. There is **no `public/robots.txt` in the repo** (`public/` contains only svgs, `logo.png`, `llms.txt`) and the only `robots.txt` artifacts are the *compiled dynamic route* under `.next/server/app/robots.txt`. So a Next route exists, yet the host is returning a different body — meaning **a `robots.txt` was placed/injected at the Hostinger document root (hPanel/Website-tools default) and the edge is serving it ahead of the Next handler.** The `x-nextjs-cache: HIT` header on HTML routes confirms Next is running, but `/robots.txt` is shadowed by the static file.
**Why it hurts:** Two failure modes. (a) **On staging today:** non-Google AI/search crawlers are *invited in* to a parked, unfinished site — duplicate/low-quality pages can be cached by Bing/AI engines despite the meta `noindex`. (b) **At cutover (the real danger):** if this static file is not removed, production will serve `Disallow: /` to Googlebot and `Allow: /` to everyone else — i.e. **you will block Google from your real site while the dynamic `app/robots.ts` (which correctly allows all) is ignored.** That is a catastrophic, silent de-indexing of the brand at launch.
**Mitigant in place:** The HTML-level second line of defense **is** working — `curl /` shows `<meta name="robots" content="noindex, nofollow">` on every staging page (verified on `/`, `/services/ai-integration`). So pages won't be *indexed* even though they can be *crawled*. Good — but robots.txt is the crawl gate and it is wrong.
**Severity:** Critical (cutover-blocking).
**Best fix:** Delete the host-level static `robots.txt` from the Hostinger document root so the Next `app/robots.ts` route is authoritative, and add a **cutover verification step** (`curl https://toshiconsulting.com/robots.txt` must equal the dynamic output). Add a defensive `public/` guard note. See Code Changes §1.
**Evidence:** `app/robots.ts:10-20`; live `curl -i /robots.txt`; `public/` listing; `find . -name robots.txt`.

### 🔴 C-2 — `NEXT_PUBLIC_SITE_URL` is host-injected → every canonical / sitemap URL / JSON-LD `@id` points to the staging domain
**What:** `SITE_URL` is read from `process.env.NEXT_PUBLIC_SITE_URL` (fallback `https://toshiconsulting.com`) in `app/layout.tsx:22`, `app/robots.ts:3`, `app/sitemap.ts:5`, `app/services/[slug]/page.tsx:9`, `app/blog/[slug]/page.tsx:20`, `app/contact/page.tsx:7`, `app/services/page.tsx:9`. The repo `.env.local` contains **only Groq/NVIDIA pipeline keys — `NEXT_PUBLIC_SITE_URL` is not set there**, yet the live site renders staging URLs everywhere:
- `<link rel="canonical" href="https://darkviolet-gerbil-901490.hostingersite.com">` (homepage)
- `<loc>https://darkviolet-gerbil-901490.hostingersite.com/…</loc>` (all 27 sitemap entries)
- Service `@id`/`url`, BlogPosting `mainEntityOfPage`, breadcrumb `item` — all staging.

So `NEXT_PUBLIC_SITE_URL` is being set **at the Hostinger build/runtime** to the staging domain.
**Why it hurts:** This is correct *for staging* but is a **single point of failure at cutover**: if the Hostinger env var is not flipped to `https://toshiconsulting.com` (and `IS_PRODUCTION_DOMAIN` only turns true for exactly `toshiconsulting.com`/`www.`), then on the real domain you will simultaneously (a) keep `noindex`, (b) keep `Disallow: /`, and (c) emit canonicals pointing at the dead staging host. Triple own-goal.
**Severity:** Critical (cutover-blocking, but a one-line env change).
**Best fix:** Make the cutover runbook set `NEXT_PUBLIC_SITE_URL=https://toshiconsulting.com` on the production host **before** DNS flip, and verify all three signals (`robots.txt`, `<meta robots>`, canonical host) in one curl pass. See Action Plan → Cutover.
**Evidence:** `.env.local` (keys only); `grep NEXT_PUBLIC_SITE_URL .env*` → empty; live canonical + sitemap `<loc>` values.

# High Priority Improvements 🟠

### 🟠 H-1 — `WebSite` `SearchAction` advertises a search feature that does not exist
**What:** `app/layout.tsx:103-107` emits `potentialAction: SearchAction` targeting `${SITE_URL}/blog?q={search_term_string}`. Confirmed non-functional: `curl /blog?q=test` returns **byte-identical** HTML to `curl /blog` (both 149,032 b), and the blog index has **no search input / no `name="q"` / no `type="search"`** (verified in `app/blog/page.tsx`, `app/blog/BlogSidebar.tsx`). The `q` param is silently ignored.
**Why it hurts:** Google's Sitelinks Searchbox uses `SearchAction`; advertising one that 404-equivalents to an unfiltered list is a schema-vs-reality mismatch that can suppress the feature and is a minor trust ding. AI engines may also surface a broken "search this site" affordance.
**Severity:** High.
**Best fix — two valid paths (genuine trade-off):**
- **(A) Remove the `SearchAction`** (5-minute fix, honest). Recommended *now* given content volume is tiny (8 posts — search adds little UX value).
- **(B) Implement client-side blog search** reading `?q=` and filtering the post list (worthwhile *after* the content build reaches 25+ posts). Then the schema becomes true.
**Recommendation:** Do (A) today; revisit (B) at ~25 posts. See Code Changes §2.
**Evidence:** `app/layout.tsx:103-107`; identical byte sizes for `/blog` vs `/blog?q=test`; no search UI in blog components.

### 🟠 H-2 — Service detail pages have no FAQ content or FAQPage schema (AI-answer gap, not a rich-result gap anymore)
**What:** `app/services/[slug]/page.tsx` emits `Service` (`:33-47`) + `BreadcrumbList` (`:49-57`) only — **no FAQs in the body and no FAQPage schema**. Confirmed on the live page: `/services/ai-integration` JSON-LD types are `Service`, `BreadcrumbList`, `Organization`, `WebSite`/`SearchAction` — no `FAQPage`. The data model already supports FAQs (`app/blog/lib.ts:17` `type FAQ`), and blog posts render them correctly (`/blog/what-is-mining-in-blockchain` shows `FAQPage`+`Question`+`Answer`).
**Important 2026 correction to the brief:** Google **deprecated FAQ rich results on 7 May 2026** (search appearance gone; rich-result report/test removed June 2026). So this is **no longer a Google "featured-snippet / rich-result" win** — do not pitch it as one. **However** FAQPage is still a valid schema, still harmless, and is **actively consumed by AI Overviews, ChatGPT, Perplexity, and Gemini** for answer extraction and citation. **Severity:** High (for AI-search + conversion; Medium for classic Google).
**Best fix:** Add a `faqs: FAQ[]` field to each `Service` in `app/data/services.ts`, render an accordion on the service page, and emit FAQPage schema **only when `faqs.length > 0`** (mirror the blog pattern at `app/blog/[slug]/page.tsx:96-104`). Content for the FAQs is a content-writer brief, not invented here. See Code Changes §3.
**Evidence:** `app/services/[slug]/page.tsx:33-57`; live JSON-LD type counts; WebSearch (Google FAQ deprecation, May 2026).

### 🟡 M-2 — Blog index & category pages lack `CollectionPage`/`ItemList` schema
**What:** `app/blog/page.tsx` and `app/blog/category/[slug]/page.tsx` render lists but emit no `CollectionPage`/`Blog`/`ItemList` JSON-LD and no `BreadcrumbList`. (Individual posts do have breadcrumbs.)

**Severity:** Medium.
**Best fix:** Emit `CollectionPage` + `ItemList` of the page's posts, plus a `BreadcrumbList` (Home › Blog [› Category]).

### 🟡 M-3 — Pagination indexability: `/blog/page/2…` in sitemap with `priority 0.6` but no `rel` semantics
**What:** `app/sitemap.ts:22-30` adds `/blog/page/{n}` pages to the sitemap. With only 8 posts and `POSTS_PER_PAGE=12` (`app/blog/lib.ts:147`), `totalPages = 1`, so **no pagination URLs exist today** (correct). But once content grows past 12 posts, paginated pages will be indexable list pages competing with the canonical `/blog`. There is no self-canonical strategy or `noindex,follow` on page 2+.
**Why it hurts:** Future thin/duplicate list pages diluting crawl budget once content scales.
**Severity:** Medium (latent — bites at >12 posts).
**Best fix:** Keep page 2+ in the sitemap *or* (preferred) self-canonicalize each paginated page to itself and let them stay `index,follow` (Google's current guidance — don't `noindex` paginated series). At minimum, ensure each `/blog/page/n` sets `alternates.canonical` to its own URL. Re-audit when content > 12 posts.

### 🟡 M-4 — `dateModified` always equals `datePublished`
**What:** `app/blog/[slug]/page.tsx:81` sets `dateModified: post.date` (same as `datePublished`). Frontmatter has only `date`.
**Why it hurts:** Freshness is a ranking/AI-citation factor; identical dates mean updates never signal freshness, and old posts look stale forever.
**Severity:** Medium.
**Best fix:** Add an optional `updated:` frontmatter field; fall back to `date` when absent. Map `dateModified` to `updated ?? date`.

### 🟡 M-5 — `priceRange: '₹₹'` is non-standard for schema
**What:** `app/contact/page.tsx:49` uses `priceRange: '₹₹'`. Google's examples use `$`-style or a numeric range; `₹₹` is unconventional and may be ignored.
**Why it hurts:** Minor — may not parse; low risk.
**Severity:** Low-Medium.
**Best fix:** Use a descriptive range like `'₹₹ — project-based'` or omit; or provide `offers`/`priceSpecification` if you publish indicative pricing.

# Code Changes

### §1 — Make the dynamic robots.txt authoritative + defensive comment (`app/robots.ts`)
The code is already correct. The fix is operational (delete the host static file) **plus** a guard so nobody adds a `public/robots.txt` later. Add this comment and keep the route as-is:

```ts
// app/robots.ts
// NOTE: This dynamic route is the ONLY source of truth for robots.txt.
// Do NOT add public/robots.txt — a static file in public/ (or one placed at the
// Hostinger document root via hPanel) will SHADOW this route at the edge and can
// invert the rules (we observed a host file blocking Googlebot + allowing all others
// on staging — the opposite of intended). Verify after every deploy:
//   curl https://<host>/robots.txt   ===   the output of this function.
import type { MetadataRoute } from 'next';

const SITE_URL = process.env.NEXT_PUBLIC_SITE_URL ?? 'https://toshiconsulting.com';
const PRODUCTION_HOSTS = ['toshiconsulting.com', 'www.toshiconsulting.com'];
const IS_PRODUCTION_DOMAIN = PRODUCTION_HOSTS.includes(new URL(SITE_URL).hostname);

export default function robots(): MetadataRoute.Robots {
  if (IS_PRODUCTION_DOMAIN) {
    return {
      rules: [{ userAgent: '*', allow: '/' }],
      sitemap: `${SITE_URL}/sitemap.xml`,
      host: SITE_URL,
    };
  }
  return { rules: [{ userAgent: '*', disallow: '/' }] };
}
```
**Action that actually fixes it:** remove the static `robots.txt` from the Hostinger document root (hPanel → File Manager / Website tools). No code change resolves an edge-served static file.
